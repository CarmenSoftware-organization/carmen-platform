## Findings by page

### BusinessUnitEdit (A4, 6/12)
- [P0][M] `canEdit` (permission gate) is threaded into InlineField groups and the is_active/is_hq toggles, but never reaches `CalculationSettingsSection`, `NumberFormatsSection`, `ConfigurationSection`, `DatabaseConnectionSection` (all receive a hardcoded `editing: true` with no `disabled`/`canEdit` prop at all), nor `BusinessUnitBrandingCard` (`editing` passed as literal `true`), nor `BusinessUnitUsersCard` (no `canEdit` prop exists on the component). A user without `cluster.update`/`cluster.create` can still type into DB connection host/port/user/password fields, change calculation method/currency, add/remove config rows, upload a new logo/avatar, and add/edit/remove BU users. The Save button (`src/pages/BusinessUnitEdit.tsx:589-601`) also has no `!canEdit` check, so those edits can be persisted. (src/pages/businessUnitEdit/BusinessUnitDocument.tsx:129, src/pages/BusinessUnitEdit.tsx:528, src/pages/BusinessUnitEdit.tsx:561, src/pages/BusinessUnitEdit.tsx:592)
- [P1][S] Page never uses the shared `PageHeader` component that every other A4 reference page uses (ClusterEdit, ReportTemplateEdit, UserEdit, UserPlatformEdit) ŌĆö per the audit rubric's shell-reuse clarification this is a P1 by definition. As a result the page has no semantic `<h1>`; the "title" is a `<button>` (`HeroName`) buried inside the hero Card, not a heading element, and there is no Edit/read toggle in the header region the contract expects. (src/pages/BusinessUnitEdit.tsx:485-491, src/pages/businessUnitEdit/BusinessUnitDocument.tsx:45-86)
- [P1][S] The page's primary status indicators (Active/Inactive, HQ) are hand-rolled pill `<button>`s with a manually-colored dot (`bg-success` / `bg-muted-foreground`), not the shared `<Badge variant="success"|"secondary">` primitive ŌĆö a direct violation of the rubric's explicit "status via `<Badge>`, never a hand-rolled colored span" rule. (src/pages/businessUnitEdit/BusinessUnitDocument.tsx:180-199)
- [P1][S] `InlineField` ŌĆö the control used for essentially every field on the page (~50 rows) ŌĆö renders its read/click-to-edit button with `px-2.5 py-1.5` on `text-sm`, giving a measured tappable height of roughly 32px, well under the 44px minimum tap-target the contract requires. Because it's one shared component, this single gap affects nearly the entire page on mobile. (src/pages/businessUnitEdit/InlineField.tsx:139-153)
- [P2][S] Required fields (`cluster_id`, `code`, `name` ŌĆö the only three fields `validateRequired()` actually enforces) show no required-field marker in their InlineField labels ("Code", "Cluster", not "Code *"), while unrelated optional-looking config fields elsewhere on the same page ("Key *", "Label *" in ConfigurationSection) do use the asterisk convention ŌĆö inconsistent and fails the A4 "required-field markers" hierarchy requirement for the fields that matter most. (src/pages/businessUnitEdit/BusinessUnitDocument.tsx:206-209 vs src/pages/businessUnitEdit/sections/ConfigurationSection.tsx:24,33)
- [P2][S] Field-level validation errors in `InlineField` show the red message text but never add `border-destructive` to the input itself ŌĆö the input's border stays `border-primary` regardless of `error`, so the contract's "validation messages `text-xs text-destructive` with `border-destructive` on the field" pairing is only half-implemented. (src/pages/businessUnitEdit/InlineField.tsx:26-27,126-136)
- [P2][S] `CollapsibleSection` renders a clickable header with a rotating chevron implying collapse/expand, but every call site in this page (`CalculationSettingsSection`, `NumberFormatsSection`, `ConfigurationSection`, `DatabaseConnectionSection`) passes `forceOpen`, which makes `isOpen` permanently `true` ŌĆö clicking the header/chevron does nothing, a misleading affordance repeated four times on the page. (src/pages/businessUnitEdit/shared.tsx:18-35, src/pages/businessUnitEdit/sections/CalculationSettingsSection.tsx:82)

### Dashboard (A2, 7/12)
- [P0][M] The `CountsRail` widget (the page's "Estate" stat card) has no loading state and its fetch failures are swallowed silently ŌĆö each per-domain `load()` call's `catch` block is empty with only a comment ("leave the domain at null ŌĆö the rail renders "ŌĆö""), so on failure the widget just sits at permanent `ŌĆö` placeholders forever with no error message, no retry affordance, and no `role="alert"`. This is exactly the "dead/placeholder widget that never resolves to real content" the A2 contract forbids. (src/pages/Dashboard.tsx:41-44, src/pages/dashboard/CountsRail.tsx:16-51)
- [P1][S] Activity verb labels (`created`/`updated`/`published`) ŌĆö the page's only status-like indicator ŌĆö are rendered as a hand-rolled colored `<span className={cn(..., verb.text)}>` driven by a local `VERB` lookup map, never wrapped in `<Badge>`, violating D3's explicit "status via `<Badge>`, never hand-rolled colored span" rule. (src/pages/dashboard/ActivityStream.tsx:9-13, 133)
- [P1][S] Loading and error regions in `ActivityStream` carry no `role="status"` / `role="alert"`, so a screen-reader user gets no notification when the stream starts loading or fails ŌĆö the skeleton is `aria-hidden` (so it announces nothing) and the error message is a plain `<div>` with no `alert` role. (src/pages/dashboard/ActivityStream.tsx:63-71, 154-168)
- [P2][S] Two touch targets fall under the 44px tappable-hit-area rule: the domain filter chips are `h-7` (28px, no padding-driven hit-area expansion) and the `CountsRail` domain rows are ~38px tall (`py-2` around a 22px icon, no extra hit padding). (src/pages/dashboard/ActivityStream.tsx:44-50, src/pages/dashboard/CountsRail.tsx:31-46)
- [P2][S] The ActivityStream "Try again" retry button calls `loadActivity()` directly with no `toast.*` feedback on success or repeated failure, contrary to A2's "refresh/retry actions use `toast.*` for feedback" requirement ŌĆö the only feedback is the inline banner re-rendering (or not). (src/pages/dashboard/ActivityStream.tsx:65-71, src/pages/Dashboard.tsx:55-62)
- [P2][M] Several spacing/typography values sit off the documented 4px scale and token set (`px-3.5`, `py-3.5`, `mt-[9px]`, `text-[10.5px]`, `text-[11.5px]`, `text-[12.5px]`, `text-[13.5px]`), producing a denser, bespoke rhythm inconsistent with the rest of the app's `gap-4`/`gap-6`/`text-xs`/`text-sm` scale. (src/pages/dashboard/CountsRail.tsx:19-44, src/pages/dashboard/ActivityStream.tsx:102-144)
