## Findings by page

### BusinessUnitEdit (A4, 11/12 — all findings below FIXED in Wave 2)
- [P0][M] `canEdit` (permission gate) is threaded into InlineField groups and the is_active/is_hq toggles, but never reaches `CalculationSettingsSection`, `NumberFormatsSection`, `ConfigurationSection`, `DatabaseConnectionSection` (all receive a hardcoded `editing: true` with no `disabled`/`canEdit` prop at all), nor `BusinessUnitBrandingCard` (`editing` passed as literal `true`), nor `BusinessUnitUsersCard` (no `canEdit` prop exists on the component). A user without `cluster.update`/`cluster.create` can still type into DB connection host/port/user/password fields, change calculation method/currency, add/remove config rows, upload a new logo/avatar, and add/edit/remove BU users. The Save button (`src/pages/BusinessUnitEdit.tsx:589-601`) also has no `!canEdit` check, so those edits can be persisted. (src/pages/businessUnitEdit/BusinessUnitDocument.tsx:129, src/pages/BusinessUnitEdit.tsx:528, src/pages/BusinessUnitEdit.tsx:561, src/pages/BusinessUnitEdit.tsx:592) **FIXED in Wave 2** (commit `7177cf4`) — `canEdit` now threads through every write surface; verified against current source.
- [P0][S] An 8th surface the audit didn't name: `Ctrl/Cmd+S` reached `handleSave` directly, so the disabled Save button was no defence on its own — a read-only user could still trigger a save via the keyboard shortcut. **Discovered and FIXED in the same commit** (`7177cf4`) — the shortcut and `handleSave` itself now both funnel through `if (!canEdit) return;`. (src/pages/BusinessUnitEdit.tsx:77, :369)
- [P1][S] Page never uses the shared `PageHeader` component that every other A4 reference page uses (ClusterEdit, ReportTemplateEdit, UserEdit, UserPlatformEdit) — per the audit rubric's shell-reuse clarification this is a P1 by definition. As a result the page has no semantic `<h1>`; the "title" is a `<button>` (`HeroName`) buried inside the hero Card, not a heading element, and there is no Edit/read toggle in the header region the contract expects. (src/pages/BusinessUnitEdit.tsx:485-491, src/pages/businessUnitEdit/BusinessUnitDocument.tsx:45-86) **FIXED in Wave 2** (commit `7177cf4`) — adopts the shared `PageHeader`, real `<h1>` confirmed in current source.
- [P1][S] The page's primary status indicators (Active/Inactive, HQ) are hand-rolled pill `<button>`s with a manually-colored dot (`bg-success` / `bg-muted-foreground`), not the shared `<Badge variant="success"|"secondary">` primitive — a direct violation of the rubric's explicit "status via `<Badge>`, never a hand-rolled colored span" rule. (src/pages/businessUnitEdit/BusinessUnitDocument.tsx:180-199) **FIXED in Wave 2** (commit `7177cf4`) — status now carried by `<Badge>` inside the toggle button; verified against current source.
- [P1][S] `InlineField` — the control used for essentially every field on the page (~50 rows) — renders its read/click-to-edit button with `px-2.5 py-1.5` on `text-sm`, giving a measured tappable height of roughly 32px, well under the 44px minimum tap-target the contract requires. Because it's one shared component, this single gap affects nearly the entire page on mobile. (src/pages/businessUnitEdit/InlineField.tsx:139-153) **FIXED in Wave 2** (commit `7177cf4`) — a `::before` hit-slop overlay stretches the tappable area to 44px while the visual box stays compact; verified against current source (`InlineField.tsx:42-44`).
- [P2][S] Required fields (`cluster_id`, `code`, `name` — the only three fields `validateRequired()` actually enforces) show no required-field marker in their InlineField labels ("Code", "Cluster", not "Code *"), while unrelated optional-looking config fields elsewhere on the same page ("Key *", "Label *" in ConfigurationSection) do use the asterisk convention — inconsistent and fails the A4 "required-field markers" hierarchy requirement for the fields that matter most. (src/pages/businessUnitEdit/BusinessUnitDocument.tsx:206-209 vs src/pages/businessUnitEdit/sections/ConfigurationSection.tsx:24,33) **FIXED in Wave 2** (commit `7177cf4`) — `required: true` now set on code/cluster_id (name lives in the header).
- [P2][S] Field-level validation errors in `InlineField` show the red message text but never add `border-destructive` to the input itself — the input's border stays `border-primary` regardless of `error`, so the contract's "validation messages `text-xs text-destructive` with `border-destructive` on the field" pairing is only half-implemented. (src/pages/businessUnitEdit/InlineField.tsx:26-27,126-136) **FIXED in Wave 2** (commit `7177cf4`) — `inputClassFor` now applies `border-destructive` on error, plus `aria-invalid`; verified against current source.
- [P2][S] `CollapsibleSection` renders a clickable header with a rotating chevron implying collapse/expand, but every call site in this page (`CalculationSettingsSection`, `NumberFormatsSection`, `ConfigurationSection`, `DatabaseConnectionSection`) passes `forceOpen`, which makes `isOpen` permanently `true` — clicking the header/chevron does nothing, a misleading affordance repeated four times on the page. (src/pages/businessUnitEdit/shared.tsx:18-35, src/pages/businessUnitEdit/sections/CalculationSettingsSection.tsx:82) **FIXED in Wave 2** (commit `7177cf4`) — `forceOpen` now renders plain non-interactive text (no cursor, no chevron, no click handler); verified against current source (`shared.tsx:24-44`).
