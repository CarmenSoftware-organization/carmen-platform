- **Task 3 — SqlWorkbench "true cancel" needs a backend endpoint (no true cancel shipped).** Traced the Run path: `sqlQueryService.executeSql` (frontend) → `ConfigSqlQueryController.execute` → `ConfigSqlQueryService.execute` (gateway, `apps/backend-gateway/src/config/config_sql-query/config_sql-query.controller.ts:97-106` and `config_sql-query.service.ts:31-65`) → `businessService.send('business.sql-query-execute', …)` over the custom HTTP RPC transport (`packages/nest-http-transport/src/http-client.proxy.ts`) → `SqlQueryController.execute` → `SqlQueryService.execute` (`apps/micro-business/src/sql-query/sql-query.service.ts:90-150`), which runs the statements inside `prisma.$transaction(...)` with `SET LOCAL statement_timeout = '30s'`. The gateway controller does `await this.service.execute(...)` and only responds afterward — it never listens for `req.on('close')`/client-disconnect, so aborting the browser's request cannot reach the gateway→microservice call at all; the microservice's Postgres transaction has no cancellation hook either. **Aborting the client HTTP request only stops the browser from waiting — the query keeps running server-side until it finishes or hits its own `statement_timeout`/transaction timeout.** Follow-up needed for a real cancel: (1) gateway `execute()` must watch `req` for disconnect and unsubscribe/abort the microservice call (the HTTP RPC transport already supports this — `HttpClientProxy.publish()`, same file, returns a teardown that calls `controller.abort()` on unsubscribe, it's just never wired to client disconnect here); (2) the microservice must actually kill the in-flight query on abort (e.g. run it on a dedicated connection and issue `pg_cancel_backend`/`pg_terminate_backend`, or accept a cancel token Prisma can honor) — today aborting the HTTP call to the microservice would only drop the response stream, not stop the already-dispatched `$transaction`. Shipped instead: `AbortController` wired to abort the request on `SqlWorkbench` unmount only (prevents a leaked request / stale state update after navigating away); no user-facing "Cancel" button, since one would lie about stopping the DB query. See `src/services/sqlQueryService.ts` (`executeSql`) and `src/pages/sqlWorkbench/SqlWorkbench.tsx` (`runSql`, `runAbortControllerRef`).
- **Task 5 — TenantMigrationManagement "true cancel" needs a backend endpoint (no true cancel shipped; this one is explicitly-by-design, not just unwired).** Traced the deploy-stream path in `../carmen-turborepo-backend-v2`: `tenantMigrationService._streamDeploy` (frontend `fetch`) → gateway `TenantMigrationsController.deployStream` (`apps/backend-gateway/src/platform/tenant-migrations/tenant-migrations.controller.ts:170-225`), which — unlike SqlWorkbench's gateway controller — **does** watch `req.on('close', () => sub.unsubscribe())`; that unsubscribe reaches `TenantMigrationsService.runDeployStream`'s `client.send(...)` (`tenant-migrations.service.ts:94-100`), which is the HTTP RPC `ClientProxy` — unsubscribing there calls the teardown from `HttpClientProxy.publish()` (`packages/nest-http-transport/src/http-client.proxy.ts:85-99`), which does `controller.abort()` on the axios request to the microservice. On the microservice side, `HttpTransportServer.streamPackets` (`packages/nest-http-transport/src/http-transport.server.ts:156-197`) itself watches `res.on('close', () => subscription.unsubscribe())` with a comment stating the intent is exactly "so a long-running observable (e.g. a prisma migrate deploy) does not keep executing after the client is gone" (`http-transport.server.ts:189-192`) — so the abort signal DOES propagate mechanically all the way to the microservice's top-level subscription. **But the actual migration handler overrides that intent on purpose.** `TenantMigrationService.deployStream` (`apps/micro-business/src/authen/tenant_migration/tenant_migration.service.ts:378-416`) has an explicit code comment: *"The inner `runBuStream` subscription is intentionally NOT torn down when the outer Observable is unsubscribed (e.g. the gateway's client disconnects). Per the design, a client disconnect must NOT abort an in-flight migration: `runBuStream` runs to natural completion (or its `spawnPrisma` timeout) and `finalize()` releases the per-BU lock then."* (`tenant_migration.service.ts:388-393`). The teardown returned at line 413 (`() => sub.unsubscribe()`) only tears down the already-completed connection-resolution observable, never the inner `runBuStream(conn)…subscribe(...)` that is actually running `spawnPrisma(['migrate', 'deploy', …])` (a real child process, killed only by its own timeout at `tenant_migration.service.ts:213-218,238-243`). `deployAllStream` (`tenant_migration.service.ts:497-580`) behaves the same way for the BU currently mid-migration: its `cancelled` flag (set via the outer teardown at line 576-578) is only checked at the TOP of the per-BU loop (line 518), so it stops the batch from STARTING the next BU but does not interrupt the one already running. **Verdict: aborting the client stream never stops an in-flight `prisma migrate deploy` — this is the backend's deliberate design, not an oversight** (contrast SqlWorkbench, where the gap was simply unwired). Grepped the whole `tenant_migration`/`tenant-migrations` domain (frontend service, gateway controller+service, microservice controller+service) for `cancel`/`abort`/`kill`: no cancel or rollback endpoint exists anywhere in this domain today — only `resolve` (`POST /api-system/tenant/migrations/:bu_id/resolve`), which marks an already-finished/stuck migration as `applied`/`rolled-back` after the fact; it is bookkeeping, not a cancel. Follow-up needed for a real cancel: (1) a dedicated cancel/rollback endpoint that (a) actually kills the in-flight `spawnPrisma` child process (the SIGKILL machinery already exists at `spawnPrisma`'s teardown, `tenant_migration.service.ts:240-243` — it would need to be reachable from an explicit cancel command, not only from unsubscribe-that-never-happens-here), and (b) since Prisma's `migrate deploy` applies migrations one-at-a-time and each one is NOT internally wrapped in a rollback-safe transaction spanning the whole batch, killing mid-migration can leave the tenant DB in a partially-migrated state — the existing `resolve` endpoint (mark applied/rolled-back) is the recovery path today and any future cancel needs to funnel into the same stuck-migration workflow, not bypass it. Shipped instead: `AbortController` (one per in-flight stream, tracked in a `Map` keyed by `bu.id` for per-row Apply and a sentinel key for Deploy all) wired to abort every outstanding request on `TenantMigrationManagement` unmount only (prevents a leaked fetch / stale state update after navigating away); no user-facing "Cancel" button, since one would tell the user they stopped an irreversible schema migration they didn't. See `src/services/tenantMigrationService.ts` (`_streamDeploy`, `deployStream`, `deployAllStream`) and `src/pages/TenantMigrationManagement.tsx` (`applyOne`, `deployAll`, `activeStreamControllersRef`).
